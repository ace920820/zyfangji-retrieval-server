# 中医方剂智能检索系统（MVP）后端开发需求文档

# 1. 项目背景

## 1.1 项目目标

构建一个基于中医典籍知识库的智能检索系统。

医生输入患者的：

* 主病主症
* 复合症状
* 舌诊
* 脉象
* 中医证型

系统从中医知识库中检索最匹配的方剂，并返回相关信息。

---

一期（MVP）目标：

> 症状 → Hybrid Search → Rerank → 返回方剂列表

MVP 仅实现：

* 本地知识文件导入
* 检索文本构建
* BM25 关键词召回
* BGE-M3 向量召回
* Hybrid Fusion
* BGE-Reranker-v2-m3 重排
* 检索接口
* 基础状态接口
* 接口文档与 Docker Compose 部署

MVP 不涉及：

* 客户 MySQL 直连或同步
* 后台管理系统
* 知识库可视化维护
* LLM 生成
* 多轮对话
* Agent
* 自动追问
* 症状标准化
* NER

说明：

客户后续可能会将数据维护在 MySQL 或其他业务系统中，但目前尚未提供数据库结构、访问方式和同步规则。因此 MVP 阶段不假设客户 MySQL 结构，优先以本地 Excel / 本地结构化文件作为知识来源完成检索链路验证。

---

# 2. 技术选型

| 模块        | 技术                 | 说明 |
| --------- | ------------------ | ---- |
| Web 框架    | FastAPI            | 提供检索接口、导入接口、状态接口和 OpenAPI 文档 |
| 本地数据读取   | Pandas / OpenPyXL  | 解析客户提供的 Excel 样例 |
| 本地元数据存储 | SQLite / JSONL     | MVP 内部保存导入记录、原始字段、索引状态；不绑定客户 MySQL |
| 向量库       | Qdrant             | 保存 BGE-M3 向量并支持向量召回 |
| Embedding | BGE-M3             | 中文语义向量模型 |
| 稀疏检索      | BM25               | 关键词召回 |
| Rerank    | BGE-Reranker-v2-m3 | 对 Top50 候选进行重排 |
| 检索策略      | Hybrid Search      | BM25 + Embedding 融合 |
| 部署        | Docker Compose     | FastAPI + Qdrant + 可选本地模型服务 |

后续如果客户明确提供 MySQL 结构，可新增 MySQL 适配层，将 MySQL 作为上游数据源之一，但不改变核心检索链路。

---

# 4. 数据结构

## 4.1 原始知识记录

MVP 不强绑定数据库表结构，而是定义一套内部 canonical record。

主要字段：

```text
entry_id

source_book

source_row

code

main_part

sub_part

main_symptom

complex_symptom

detail_symptom

alias

tongue

pulse

syndrome

formula

therapy

tcm_disease

western_disease

pathology

cause

contraindication

effect

source

origin_text

human_body_image

retrieval_text

raw_record

create_time

update_time
```

说明：

* `entry_id` 是系统内部稳定 ID，用于检索结果回溯和索引维护。
* `code` 保留客户数据中的编码，但不强依赖其完整性。
* `formula` 保留原始推荐方剂文本；如一行包含多个方剂，MVP 先保留原文并可标记为 `needs_review`。
* `raw_record` 保存原始行内容，便于后续排查和字段映射调整。

---

## 4.2 检索文本

由以下字段拼接：

```text
主干部位

分支部位

主病主症

复合症

细分主症

同症异名

舌诊

脉象

中医证型
```

例如：

```text
部位：
头

主症：
头痛

复合症：
发热
恶风
无汗
身痛

同义词：
头疼
怕风

舌诊：
舌苔薄白

脉象：
浮紧脉

证型：
太阳伤寒证
```

记作：

```text
retrieval_text
```

# 5. 检索流程设计

## Step1 Query 构建

用户输入：

```json
{
  "main_symptom": "头痛",
  "symptoms": [
    "发热",
    "恶风",
    "无汗"
  ],
  "tongue": "舌苔薄白",
  "pulse": "浮脉",
  "syndrome": "太阳伤寒证",
  "topk": 10
}
```

构造成：

```text
主症：
头痛

症状：
发热
恶风
无汗

舌诊：
舌苔薄白

脉象：
浮脉

证型：
太阳伤寒证
```

---

## Step2 BM25 召回

目的：

关键词召回。

例如：

```text
头痛

发热

麻黄汤

太阳病
```

返回：

```text
Top50
```

---

## Step3 BGE-M3 向量召回

目的：

语义召回。

例如：

```text
头疼

≈

头痛


怕风

≈

恶风


发烧

≈

发热
```

返回：

```text
Top50
```

---

## Step4 Hybrid Fusion

融合：

```text
BM25 Top50

+

Embedding Top50
```

推荐优先使用：

```text
RRF（Reciprocal Rank Fusion）
```

也可配置为 Weighted Score：

```text
score = 0.3 * bm25_score + 0.7 * embedding_score
```

输出：

```text
Top50 候选
```

---

## Step5 Rerank

模型：

```text
BGE-Reranker-v2-m3
```

输入：

```text
Query + Top50候选
```

输出：

```text
Top10
```

作为最终结果。

# 6. 后端接口设计

## 6.1 导入知识库

### 接口

```text
POST /api/import
```

### 功能

上传或读取本地 Excel。

流程：

```text
上传 / 指定本地 Excel

↓

解析

↓

字段映射与校验

↓

生成 canonical record

↓

生成 retrieval_text

↓

写入本地元数据存储

↓

BGE-M3 Embedding

↓

写入 Qdrant

↓

构建 BM25 索引

↓

完成
```

返回示例：

```json
{
  "source": "伤寒论",
  "total_rows": 1329,
  "valid_rows": 1248,
  "skipped_rows": 3,
  "warning_count": 12,
  "indexed_count": 1248,
  "index_version": "20260614-001"
}
```

---

## 6.2 检索接口

### 接口

```text
POST /api/search
```

请求：

```json
{
  "main_symptom": "头痛",
  "symptoms": [
    "发热",
    "恶风",
    "无汗"
  ],
  "tongue": "舌苔薄白",
  "pulse": "浮脉",
  "syndrome": "太阳伤寒证",
  "topk": 10
}
```

返回：

```json
{
  "results": [
    {
      "entry_id": "shl-000001",
      "code": "1001",
      "formula": "麻黄汤",
      "therapy": "辛温解表",
      "score": 0.92,
      "score_type": "retrieval_relevance",
      "main_symptom": "头痛",
      "complex_symptom": "头痛+发热+恶风+无汗",
      "tongue": "舌质淡红，苔薄白",
      "pulse": "浮紧脉",
      "syndrome": "太阳伤寒证",
      "source": "伤寒论第35条",
      "origin_text": "太阳病，头痛发热……"
    }
  ]
}
```

说明：

* `score` 表示检索相关性，不表示医学置信度或处方概率。
* `entry_id` 是系统内部 ID，用于回溯原始知识记录。
* `code` 如客户数据缺失，可为空。

# 7. MVP 后升级规划

MVP 完成后，可逐步增加以下能力。

---

## 7.1 客户 MySQL / 业务系统数据同步

当客户明确提供数据库结构、访问方式和同步规则后，可新增 MySQL 适配层。

可能能力：

```text
从客户 MySQL 拉取知识数据

定时同步

手动同步

增量同步

变更后重建索引
```

注意：

MVP 阶段不依赖该能力。

## 7.3 症状标准化

目标：

统一症状表达，提高召回率。

例如：

```text
头疼

↓

头痛


发烧

↓

发热


怕风

↓

恶风
```

数据来源：

优先使用：

```text
同症异名
```

字段。

后续可新增：

```text
t_alias
```

表进行维护。

---

## 7.4 医学 NER

支持：

```text
我最近头疼，

发烧，

怕风，

舌苔发白。
```

自动抽取：

```json
{
  "main_symptom": "头痛",
  "symptoms": [
    "发热",
    "恶风"
  ],
  "tongue": "舌苔薄白"
}
```

再进入检索。

---

## 7.5 LLM 生成

升级：

```text
检索

↓

Rerank

↓

LLM

↓

生成推荐理由
```

例如：

```text
推荐：

麻黄汤

原因：

患者出现：

头痛、发热、恶风、无汗，

符合《伤寒论》中太阳伤寒证的表现，

因此推荐麻黄汤。
```

---

## 7.6 多轮对话

支持：

```text
头痛

↓

请问是否伴随发热？

↓

有

↓

请问是否怕风？

↓

怕

↓

推荐麻黄汤
```

升级为完整的中医智能问诊系统。

# 8. 性能要求

数据规模：

```text
一期：
1000~2000条

后续：
约200本典籍
20~30万条以内
```

响应时间：

```text
P50 < 500ms

P95 < 1s
```

说明：

MVP 阶段响应时间以已完成索引构建后的 `/api/search` 为准，不包含首次导入、Embedding 生成和索引重建耗时。
