<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Java backend -> response contract | Patient queries produce doctor-facing evidence and score semantics consumed by another service. |
| Search pipeline -> evidence projection | Internal candidate scores and source records become externally visible JSON. |
| Search route -> error envelope | Internal failures cross to Java clients as stable machine-readable error codes. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-03-01 | Information Disclosure | `api/routes/search.py` | mitigate | Do not log raw patient request bodies or raw provider errors containing patient text. |
| T-03-03-02 | Safety/Misuse | response score fields | mitigate | Keep `score_semantics` in every response and forbid `confidence`, `diagnosis_probability`, and `prescription_certainty` fields. |
| T-03-03-03 | Repudiation | API error responses | mitigate | Translate search failures into `SearchErrorEnvelope` with stable codes: `index_not_ready`, `vector_store_unavailable`, `reranker_unavailable`, and `search_internal_error`. |
| T-03-03-04 | Denial of Service | `/api/search` | mitigate | Preserve request `topk <= 50`, recall Top50, and bounded reranker candidate list from earlier plans. |
| T-03-03-05 | Tampering | formula mapping | mitigate | Return `formula_mapping_status` and nullable `formula_code`; do not invent formula codes or fetch business formulary data. |
| T-03-03-06 | Spoofing | stale/partial index response | mitigate | Include `index_version` and `metadata_version` in metadata and continue to rely on SearchService active-index checks. |
</threat_model>
