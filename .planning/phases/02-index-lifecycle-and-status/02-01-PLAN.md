<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| local metadata -> embedding provider | `retrieval_text` leaves local model code only through a configured provider implementation; this plan ships deterministic local provider behavior and validation. |
| indexer -> Qdrant | Dense vectors and payload metadata cross into Qdrant; payload must exclude secrets and provider credentials. |
| tests -> vector repository | Tests must inject fake/local Qdrant behavior so no unauthenticated network service is required. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01-01 | Information Disclosure | `build_qdrant_payload` | mitigate | Payload includes canonical `KnowledgeEntry` fields needed for retrieval/evidence only; never include API keys, config secrets, or environment dumps. |
| T-02-01-02 | Tampering | `validate_embedding_batch` / `QdrantVectorIndex.upsert_entries` | mitigate | Validate vector count and vector_size before any upsert so malformed provider output cannot create a partial vector collection. |
| T-02-01-03 | Denial of Service | embedding provider boundary | accept | This plan has no HTTP mutation endpoint and deterministic tests do not call external providers; provider rate/timeout policy belongs to the selected real provider implementation. |
| T-02-01-04 | Spoofing | Qdrant alias activation | mitigate | Alias activation is exposed only through repository methods for later CLI lifecycle use; no unauthenticated HTTP rebuild/activation endpoint is planned in Phase 2. |
</threat_model>
