<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| external HTTP client -> FastAPI status API | Public or Java-backend callers can read health/status JSON. |
| FastAPI status API -> SQLite local state | Read-only API code loads active/latest build records from local SQLite metadata. |
| deployment probe -> readiness | Readiness controls whether the service is treated as able to serve indexed operations later. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-03-01 | Information Disclosure | `/status` and `/health/ready` | mitigate | Return provider/model identifiers, counts, versions, and last_error only; do not return API keys, raw source records, raw patient data, or environment dumps. |
| T-02-03-02 | Denial of Service | FastAPI routes | mitigate | Status/readiness are read-only and must not call embedding providers or rebuild indexes. |
| T-02-03-03 | Tampering | HTTP surface | mitigate | This plan intentionally adds no high-risk network mutation endpoint; rebuild remains CLI/local in Phase 2, so unauthenticated HTTP clients cannot trigger expensive or state-changing index work. |
| T-02-03-04 | Repudiation | failure visibility | mitigate | Surface `last_error`, latest build time, and readiness false states from SQLite so import/rebuild failures are visible instead of silently passing. |
| T-02-03-05 | Spoofing | readiness response | mitigate | Readiness is derived from SQLite active state plus count consistency; no caller-provided query parameter can force `ready=true`. |
</threat_model>
