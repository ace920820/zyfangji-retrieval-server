<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| local CLI -> filesystem paths | Operator-provided `--db-path` and index roots are local paths used for SQLite and BM25 files. |
| local metadata -> index artifacts | Imported KnowledgeEntry records become Qdrant payloads and BM25 corpus metadata. |
| lifecycle -> provider/Qdrant/BM25 | Failures in external provider or vector store must not mutate active SQLite state. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-02-01 | Tampering | `IndexLifecycleService.rebuild` | mitigate | Record build as `building`, write version-specific Qdrant/BM25 artifacts, validate counts, then activate in SQLite only after validation succeeds. |
| T-02-02-02 | Repudiation | `SQLiteIndexStateStore` | mitigate | Persist `index_builds.status`, timestamps, counts, provider/model identifiers, and `last_error` for every build attempt. |
| T-02-02-03 | Denial of Service | rebuild operation | mitigate | Keep rebuild as local CLI operation in this phase; do not expose unauthenticated HTTP rebuild mutation. |
| T-02-02-04 | Information Disclosure | CLI output | mitigate | CLI prints counts, versions, provider/model identifiers, and errors; it must not print raw source rows or provider secrets. |
| T-02-02-05 | Spoofing | active index source | mitigate | SQLite `active_index` is the unified source for active version; Qdrant alias can mirror vector activation but cannot be the only active-state source. |
</threat_model>
